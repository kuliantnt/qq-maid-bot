# PRD：可配置上下文模块加载

## 状态

Planned

本文档用于记录后续可能的演进方向，当前版本暂不实施。

## 1. 背景

当前运行时支持从配置目录加载固定提示词，并可选加载额外的上下文文件。

这种方式结构简单、行为明确，适合规模较小的配置。但随着上下文内容逐渐增加，单文件全量加载可能带来以下问题：

* 无关内容随每次请求重复注入；
* 输入上下文持续增长；
* 不同领域内容难以独立维护；
* 配置变更的影响范围较大；
* 后续扩展检索策略时缺少统一抽象。

因此，计划评估一种可配置的上下文模块加载机制，使固定规则与按需内容能够分别管理。

## 2. 目标

* 保留现有固定提示词加载方式；
* 支持将附加上下文拆分为多个独立模块；
* 根据当前请求选择需要加载的模块；
* 控制单次加载的模块数量和内容规模；
* 保持加载结果可预测、可观测；
* 为后续扩展其他匹配策略预留接口；
* 尽量保持现有配置兼容。

## 3. 非目标

初始版本暂不考虑：

* 外部向量数据库；
* embedding 生成；
* 复杂语义检索；
* 自动生成或修改上下文内容；
* 在线配置管理界面；
* 多级递归依赖；
* 概率触发和复杂规则引擎。

首个实现如有需要，应优先采用简单、确定性的本地配置方案。

## 4. 设计原则

### 4.1 固定规则与附加上下文分离

固定提示词继续承载全局行为规则和基础约束。

可选上下文模块用于承载只在特定请求中需要的补充信息。

两者应保持独立，避免附加模块改变基础运行行为。

### 4.2 配置优先

模块定义、文件位置、触发条件和优先级应由配置文件描述，而不是硬编码在业务逻辑中。

### 4.3 确定性优先

初始版本应尽量保证相同输入产生相同的模块选择结果，方便测试和排障。

### 4.4 渐进式演进

模块内容与模块选择逻辑应解耦。

未来可以替换匹配实现，而不要求重新组织现有 Markdown 内容。

## 5. 建议目录结构

```text
runtime/config/
├── prompts/
│   ├── system.md
│   ├── mode_rules.md
│   └── session_context.md
│
├── context/
│   ├── core.md
│   ├── topic_a.md
│   ├── topic_b.md
│   └── topic_c.md
│
├── context.toml
└── context.example.toml
```

其中：

* `prompts/` 保存固定提示词；
* `context/` 保存可选上下文模块；
* `context.toml` 描述模块加载规则；
* `.example` 文件提供公开配置示例。

目录和文件名称仅为建议，最终以实现阶段的代码结构为准。

## 6. 配置示例

```toml
version = 1

[settings]
max_entries = 3
max_chars = 12000

[[entries]]
id = "core"
file = "context/core.md"
always = true
priority = 100

[[entries]]
id = "topic_a"
file = "context/topic_a.md"
keywords = ["keyword-a", "keyword-b"]
priority = 80

[[entries]]
id = "topic_b"
file = "context/topic_b.md"
keywords = ["keyword-c", "keyword-d"]
priority = 70
```

字段说明：

* `id`：模块唯一标识；
* `file`：模块内容文件；
* `always`：是否始终加载；
* `keywords`：可选触发关键词；
* `priority`：多个模块同时命中时的排序依据；
* `max_entries`：单次最多加载的可选模块数；
* `max_chars`：单次加载内容的最大字符预算。

## 7. 建议加载流程

```text
接收请求
    ↓
加载固定提示词
    ↓
加载常驻上下文模块
    ↓
根据当前输入匹配可选模块
    ↓
按优先级排序
    ↓
应用数量和内容预算
    ↓
组装最终模型上下文
```

初始版本可以仅使用当前用户输入参与匹配。

是否需要结合最近若干轮会话，可在实现阶段进一步评估。

## 8. 匹配策略

首个版本建议采用简单关键词匹配：

* 任意关键词命中即可激活模块；
* 同一模块单次只加载一次；
* 多个模块命中时按优先级排序；
* 超出数量限制的模块不加载；
* 超出内容预算时跳过低优先级模块；
* 不自动扫描配置目录中的全部文件；
* `.example` 文件不得进入运行时上下文。

未来可在保持统一接口的前提下增加：

* 标签匹配；
* 关键词权重；
* 排除规则；
* 全文检索；
* 语义检索；
* 混合召回与重排。

## 9. 兼容性

现有单文件附加上下文配置应暂时保留。

建议兼容策略：

1. 配置模块索引时，使用新的模块加载机制；
2. 未配置模块索引时，保持当前单文件加载行为；
3. 两种配置同时存在时，应定义明确的优先级并输出提示；
4. 暂不删除现有配置项。

具体环境变量名称和迁移周期由实现阶段确定。

## 10. 校验与错误处理

启动阶段建议校验：

* 配置文件是否可解析；
* 模块 ID 是否唯一；
* 文件路径是否有效；
* 常驻模块是否存在且非空；
* 文件路径是否位于允许的配置目录内；
* 数量和字符预算是否有效。

错误信息应包含具体配置项或文件路径，方便定位问题。

## 11. 可观测性

建议通过 debug 日志记录：

* 命中的模块 ID；
* 常驻模块和动态模块数量；
* 最终加载字符数；
* 因数量或预算限制被跳过的模块；
* 当前使用模块模式还是兼容模式。

日志不应输出模块正文。

## 12. 安全要求

* 用户输入不能直接指定任意本地文件；
* 模块路径必须限制在允许的配置目录内；
* 防止通过相对路径访问配置目录之外的文件；
* 上下文正文不得默认写入运行日志；
* 仓库仅提供通用示例配置；
* 部署方负责管理实际运行内容。

## 13. 验收标准

首个版本完成时应满足：

1. 固定提示词加载行为保持不变；
2. 支持常驻上下文模块；
3. 支持按关键词加载可选模块；
4. 支持模块优先级；
5. 支持模块数量限制；
6. 支持内容字符预算；
7. 支持清晰的配置错误提示；
8. 支持查看模块命中日志；
9. 保持现有单文件配置兼容；
10. 不引入外部数据库或检索服务；
11. 不自动加载未声明文件；
12. 对现有请求接口保持兼容。

## 14. 后续方向

后续是否引入全文检索或语义检索，应根据实际规模和匹配质量决定。

可能的演进顺序：

```text
固定附加文件
    ↓
配置化上下文模块
    ↓
关键词与标签匹配
    ↓
本地全文检索
    ↓
可选语义检索
```

本设计的重点不是提前确定最终检索技术，而是先建立稳定的内容模块和加载边界。
